# SecureTrash CLI — дизайн (v1)

Дата: 2026-06-17
Статус: approved (дизайн), ожидает review спеки → writing-plans

## Проблема

Обычное удаление через Корзину не стирает данные — файлы восстановимы (Disk Drill,
PhotoRec). Распространённый «народный» совет — `rm -P` (перезапись N раз). **На
современных Mac (SSD + APFS) это не даёт гарантий** из-за wear leveling, copy-on-write
и TRIM: перезапись «поверх» физически попадает в другие ячейки, исходный plaintext
остаётся в неадресуемой области. Apple удалила `srm` и «Безопасную очистку Корзины» в
OS X 10.11 именно поэтому.

Большинство публичных «secure-delete» инструментов на GitHub это игнорируют и дают
пользователю ложное чувство защищённости.

## Цель продукта

Честный и **реально работающий** CLI для безопасного удаления файлов на macOS.
Дифференциатор — продукт не врёт про SSD: объясняет ограничения перезаписи и даёт метод
с математической гарантией (crypto-shred через зашифрованный контейнер) + настаивает на
FileVault как базовой защите.

## Не-цели (YAGNI для v1)

- GUI / Automator-droplet.
- Linux / Windows.
- Настраиваемое число проходов перезаписи (на SSD бессмысленно).
- Восстановление/undelete.

## Технологии

- **Чистый Bash**, только нативные macOS-утилиты: `fdesetup`, `hdiutil`, `diskutil`,
  `rm`, `dd`. Ноль рантайм-зависимостей → ставится одной строкой и полностью
  аудируется глазами (требование к security-инструменту).
- Совместимость: macOS 10.15+.
- Дистрибуция: Homebrew tap (`brew install Di-kairos/tap/securetrash`) + `curl | bash`
  инсталлятор.

## Архитектура

Один исполняемый скрипт `securetrash` с диспетчером подкоманд. Каждая подкоманда —
отдельная bash-функция с одной ответственностью; общие хелперы (вывод диска,
определение SSD/APFS, цветной вывод, подтверждения) — в начале файла.

Команда `securetrash`, короткий алиас `sectrash`.

### Подкоманды (v1)

| Команда | Назначение |
|---|---|
| `check` | Аудит окружения: FileVault (`fdesetup status`), тип диска SSD/HDD (`diskutil info`), APFS, TRIM → честный вердикт о достижимых гарантиях. Read-only. |
| `setup` | Создаёт `~/SecureTrash`, ставит алиас в `~/.zshrc`, проверяет FileVault, громко предупреждает если выключен. Идемпотентна. |
| `empty` | Опустошает `~/SecureTrash`: `chmod -R u+w` → best-effort overwrite → удаление. Честный вывод по типу диска. |
| `shred <path>...` | То же для произвольного файла/папки вне корзины. Требует подтверждения. |
| `vault create\|open\|close\|destroy` | Зашифрованный sparsebundle-контейнер (AES-256, `hdiutil`). Crypto-shred, реально работающий на SSD. |

### Поведение по типу диска

- **SSD/APFS:** `empty`/`shred` делают best-effort overwrite, но в выводе честно
  говорят: «реальная гарантия — FileVault; для секретов используй `vault`».
- **HDD:** `empty`/`shred` сообщают, что 3× перезапись здесь эффективна.

### vault — детали

```
create  → hdiutil create -size <N> -type SPARSEBUNDLE -fs APFS \
                 -encryption AES-256 -stdinpass -volname SecretVault <path>
open    → hdiutil attach <path> -stdinpass        # монтирует
close   → hdiutil detach /Volumes/SecretVault     # данные снова зашифрованы
destroy → detach + rm -rf <container>             # crypto-shred всего содержимого
```

**Гарантия:** данные внутри контейнера с самого начала лежат только зашифрованными;
plaintext не касается голых блоков. `destroy` оставляет шифрошум без ключа — физическое
расположение блоков на SSD неважно.

**Честное ограничение (проговаривается в `check`/README/выводе vault):** vault —
**превентивная** защита. Файл, который уже лежал снаружи plaintext-ом, vault задним
числом не стирает; для «уже снаружи» единственная реальная защита — FileVault на весь
диск.

## Обработка ошибок

- Не macOS → выход с понятным сообщением.
- `destroy`/`shred` → обязательное явное подтверждение (ввод `yes`), флаг `--yes` для
  скриптов.
- FileVault off в `setup`/`check` → не блок, но громкое предупреждение.
- Несуществующий путь, отсутствие прав → внятная ошибка, ненулевой код возврата.
- `vault open` без существующего контейнера → подсказка про `create`.

## Тестирование

- `bats` (bats-core): парсинг аргументов, диспетчер подкоманд, `check` на mock-выводе
  `diskutil`/`fdesetup`, ветка «не macOS». **Без реального удаления** в тестах.
- Smoke-цель: `vault create/open/close/destroy` на временном контейнере 10 МБ в CI.

## Критерии успеха

1. `securetrash check` честно классифицирует машину и выдаёт корректный вердикт.
2. `vault create→open→write→close→destroy` уничтожает контейнер; данные неизвлекаемы.
3. `shellcheck` без ошибок; `bats` зелёный в CI.
4. README не содержит ложных утверждений про гарантии перезаписи на SSD.
5. Установка одной строкой (Homebrew или curl) работает на чистой macOS 10.15+.
