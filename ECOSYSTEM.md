# SecureTrash Ecosystem — Developer Brief

> Карта для разработки. Этот документ объясняет **видение экосистемы** и даёт
> **спецификации** на следующие инструменты. Читается как единое целое: сначала
> философия (она же — главное ограничение), потом конкретные tool-спеки.
>
> Статус: `securetrash` готов (v0.3.0). Дальше строим соседей вокруг него.

---

## 1. Философия (это закон, а не лозунг)

Всё семейство держится на **одном принципе SecureTrash**: честный аудит вместо
security theater. Любой новый инструмент обязан ему соответствовать, иначе он не
часть экосистемы.

Три правила, которые проверяются в каждом PR:

1. **Один инструмент — одна задача.** Никаких комбайнов. Если утилита делает два
   дела — это две утилиты.
2. **Опора на нативные примитивы, ноль зависимостей.** Чистый Bash; на macOS —
   `launchd`, `hdiutil`, `mdutil`, `tmutil`, `diskutil`. Если без рантайма не
   обойтись — обосновать в PR.
3. **Честно про пределы.** У каждой утилиты в README есть секция
   `Scope & limitations`, где прямо написано, чего она **не** гарантирует. Если
   фича создаёт ложное чувство безопасности — её не выпускаем или громко
   предупреждаем.

Анти-цель: «продать перезапись как гарантию». Мы так не делаем — об этом весь
SecureTrash.

---

## 2. Видение: жизненный цикл секрета

Экосистема покрывает не «удаление файла», а **полную жизнь секрета** (seed,
пароль, приватный ключ, чувствительный текст). Каждый инструмент — один шаг.

```
                 ┌─────────────────┐
                 │   1. ghostdraft │  написать без следов
                 └─────────────────┘
        5. securetrash ◄── СЕКРЕТ ──► 2. vault
            уничтожить    seed/key     положить в сейф
                 ▲                          │
                 │                          ▼
            4. panic  ◄──────────────  3. vaultwatch
          спрятать по тревоге        защитить, пока открыт
```

| # | Инструмент    | Шаг жизни секрета            | Статус   |
|---|---------------|------------------------------|----------|
| — | `securetrash` | хранить (vault) + уничтожить | ✅ готов  |
| 1 | `ghostdraft`  | написать/просмотреть         | спека    |
| 2 | `vault`       | положить в сейф              | ✅ (часть securetrash) |
| 3 | `vaultwatch`  | защитить, пока сейф открыт    | спека (приоритет) |
| 4 | `panic`       | мгновенно спрятать            | спека    |
| 5 | `seedsplit`*  | распределить (опционально)    | идея     |

\* `seedsplit` (Shamir/SLIP-39) — кандидат, не обязателен для MVP экосистемы.

## 3. Порядок работы (для Claude Code)

**Приоритет №1 — `vaultwatch`.** Самый логичный сосед: закрывает дыру, про
которую `securetrash` сам предупреждает в README (mounted-vault leak surface).
Технически самый простой, переиспользует хуки `vault open/close`.

**Приоритет №2 — `panic`.** Дёшев: переиспользует close-логику из `vaultwatch`.

`ghostdraft` и `seedsplit` — после, они работают с живым plaintext/ключами и
требуют больше аккуратности.

Общие требования к каждой утилите:
- структура и стиль репозитория как у `securetrash` (Bash, `shellcheck`-clean,
  тесты, EN-вывод по умолчанию + `ST_LANG=ru`);
- интеграция с экосистемой через общий `check`-движок (см. §7);
- секция `Scope & limitations` в README обязательна.

## 4. Спека: `vaultwatch` (приоритет)

### Назначение
Демон-сторож, активный **только пока vault примонтирован**. Закрывает каналы, по
которым открытый plaintext может утечь (Spotlight, Time Machine, cloud sync), и
восстанавливает всё при закрытии.

### Жизненный цикл
- Стартует из хука `securetrash vault open`.
- Завершается из хука `securetrash vault close` (или по `--ttl`).
- Фоновый режим — через `launchd` (LaunchAgent).

### Что делает
| Действие | Реализация | Восстановление при close |
|----------|------------|--------------------------|
| Блокирует индексацию vault | `mdutil -i off /Volumes/SecretVault` | `mdutil -i on` |
| Исключает vault из Time Machine | `tmutil addexclusion <mount>` | `tmutil removeexclusion` |
| Детект активных cloud-демонов | поиск процессов Dropbox/iCloud/OneDrive/Google Drive + проверка, не лежит ли mount внутри их синк-папок | предупреждение, не «магия» |
| Авто-выход по таймеру | `--ttl 30m` → закрывает vault сам | — |

### Отчёт при закрытии (пример)
```
vaultwatch — session report
  duration:        24m 18s
  Spotlight:       indexing was OFF for /Volumes/SecretVault ✓
  Time Machine:    excluded, no local snapshot taken ✓
  cloud daemons:   Dropbox active — vault was OUTSIDE its sync folder ✓
  swap:            NOT addressed (see limitations)
```

### ⚠️ Честные границы (обязательны в README)
- **Swap не закрывается на лету.** Если во время сессии была memory pressure,
  фрагменты plaintext могли уйти в swap и остаться там до перезаписи. Говорим об
  этом прямо; не пишем «утечек нет».
- **Уже снятые локальные снапшоты.** `addexclusion` не удаляет снапшоты, снятые
  до старта демона. Проверять `tmutil listlocalsnapshots /` и сообщать в отчёте.
- **`--ttl` упирается в открытые файлы.** `hdiutil detach` не размонтирует том с
  открытыми дескрипторами. Fallback: `lsof` → предупредить; `detach -force`
  только с явным подтверждением (force = риск повреждения данных).
- **Cloud-детект эвристичен.** Формулировка «обнаружены демоны X, Y; vault вне их
  синк-папок», а не обещание телепатии.

### Чего НЕ делаем
Не чистим swap, не редактируем настройки облаков пользователя, не удаляем чужие
бэкапы. Только обратимые исключения на время сессии.

## 5. Спека: `panic`

### Назначение
Kill-switch на один шаг. Сценарий: граница / принуждение / «кто-то идёт». Одной
командой (или хоткеем через `launchd`) спрятать и запереть всё.

### Что делает
1. Закрывает все открытые vaults (`securetrash vault close` по списку).
2. `hdiutil detach -force` для томов vault; размонтирует внешние диски.
3. Очищает буфер обмена (`pbcopy </dev/null`).
4. Блокирует экран.
5. Опционально (`--hard`): прибивает cloud-демоны, чистит «Recent items».

### ⚠️ Честные границы
- `panic` **прячет и запирает**, но **не уничтожает** и **не чистит swap**. Так и
  пишем. Для уничтожения — `securetrash`.
- `detach -force` при открытых файлах может повредить данные — это осознанный
  trade-off режима паники (спрятать важнее), пользователь должен это знать.

### Чего НЕ делаем
Не имитируем «полное стирание за секунду» — это была бы ложь.

## 6. Спека: `ghostdraft` (после vaultwatch/panic)

### Назначение
Эфемерный черновик: написать/просмотреть чувствительный текст (seed, пароль,
ключ) так, чтобы после закрытия следов не осталось ни в одном из обычных мест
(`~/.*_history`, swap, tmp, recent docs, editor backups).

### Команды
| Команда | Поведение |
|---------|-----------|
| `ghostdraft new` | временный файл **внутри открытого vault** (предпочтительно) или на RAM-диске; открывает `$EDITOR`/nano; по выходу — `securetrash shred`, чистка editor-истории |
| `ghostdraft pipe` | читает stdin, печатает в терминал, **на диск не пишет ничего** (`pbpaste \| ghostdraft pipe`) |
| `--clipboard` | кладёт в буфер и чистит через N сек — **по умолчанию ВЫКЛ, с предупреждением** |

### ⚠️ Честные границы — здесь легко скатиться в снейкойл
- **`/dev/shm` на macOS НЕ существует** (это Linux). На macOS «временный файл» в
  `/tmp`/`$TMPDIR` лежит на APFS, т.е. на диске. Настоящая память — только
  RAM-диск: `hdiutil attach -nomount ram://<sectors>` + `diskutil erasevolume`.
  Реализуем честно или помечаем in-memory-режим как **Linux-only**.
- **`--clipboard` для seed-фразы опасен.** Буфер перехватывают clipboard-менеджеры,
  а macOS Universal Clipboard **синхронит его в iCloud на другие устройства**.
  Авто-очистка не отменяет уже снятую копию. По умолчанию выключено + явное
  предупреждение.
- **`pipe` безопаснее всего**, но остаётся scrollback терминала в памяти и риск
  скриншота/подглядывания.
- **Editor backups:** чистить vim `.swp`, `.un~` **и `viminfo`** (хранит регистры,
  последний yank, историю поиска — утечка мимо `.swp`), nano backup, VSCode
  workspace history.

### Чего НЕ делаем
Не обещаем «ноль следов» там, где ОС физически может оставить копию (swap,
scrollback) — перечисляем эти места честно.

## 7. Общая интеграция

- **Единый `check`-движок.** Вынести аудит-логику `securetrash check` в общий
  модуль, чтобы `vaultwatch`/`panic`/`ghostdraft` сообщали состояние в одном
  формате и с одной честностью.
- **Хуки vault.** `securetrash vault open/close` должны уметь дёргать
  pre/post-хуки — через них стартует/гасится `vaultwatch`.
- **Единый бренд.** Префикс `secure*`/общее имя, общий тон вывода, общий
  `--yes` для необратимых операций (как у `vault destroy`).
- **Кроссплатформа.** Каждая спека — с разделом «Windows-эквивалент»
  (BitLocker/VSS/pagefile/OneDrive), даже если порт идёт во вторую очередь.

## 8. Definition of Done (на каждый инструмент)

- [ ] Делает ровно одну задачу.
- [ ] Чистый Bash, ноль внешних зависимостей (или обоснование в PR).
- [ ] `shellcheck` без ошибок; тесты на логику.
- [ ] EN-вывод по умолчанию, `ST_LANG=ru`.
- [ ] README с обязательной секцией `Scope & limitations` (что НЕ гарантируем).
- [ ] Необратимые операции требуют `yes`/`--yes`.
- [ ] Раздел «Windows-эквивалент» в README.
- [ ] Не создаёт ложного чувства безопасности — проверено вручную.
