## Проблема

Когда ты удаляешь файл и очищаешь Корзину, macOS лишь помечает место как свободное.
Сами данные остаются на диске, пока их не перезапишут. Бесплатные утилиты вроде
**Disk Drill** или **PhotoRec** спокойно восстанавливают такие «удалённые» файлы.
Корзина — это не удаление, это «спрятать с глаз».

## ⚠️ Честная правда про SSD

> На современных дисках совет «перезаписать файл N раз» (`rm -P`, старый `srm`,
> «Надёжно очистить Корзину») **НЕ гарантирует** стирания данных.
>
> Контроллер SSD сам решает, в какие физические ячейки писать: **wear leveling**
> размазывает записи по диску, **copy-on-write** в APFS пишет новые версии в
> другое место, а **TRIM** освобождает блоки в фоне по своему графику. Команда
> «перезаписать этот файл» не достаёт до тех ячеек, где реально лежали твои байты.
>
> Именно поэтому Apple **удалила** `srm` и пункт «Надёжно очистить Корзину»
> ещё в OS X 10.11 El Capitan — они создавали ложное чувство безопасности.
>
> Большинство «секьюрных шреддеров» об этом молчат и продают перезапись как
> гарантию. **Мы — нет.**

## Что реально защищает

1. **FileVault** — полнодисковое шифрование. База всего. Если диск зашифрован,
   «удалённые» блоки — это шифротекст без ключа. Это твой главный и обязательный слой.
2. **Crypto-shred через `vault`** — храни секреты в зашифрованном контейнере
   (AES-256) с самого начала, а потом уничтожь контейнер вместе с ключом.
   При стойком пароле и без оставшихся копий/бэкапов/снимков данные фактически
   недоступны, где бы их блоки физически ни лежали на SSD.

Vault **превентивен**: он защищает только то, что ты создал или положил внутрь
контейнера. Он не может задним числом стереть plaintext, который уже полежал на
диске незашифрованным, — для этого нужен FileVault.

## Установка

**Рекомендуется — Homebrew** (формула пиннит релизный тег и проверяет его SHA256):

```bash
brew install Di-kairos/tap/securetrash
```

### Установка одной строкой через curl

Установщик тянет бинарь **и `SHA256SUMS` с релизного тега** (а не с подвижной ветки)
и проверяет контрольную сумму **до** установки — fail-closed при любом несовпадении.
Быстрая форма:

```bash
curl -fsSL https://github.com/Di-kairos/securetrash/releases/latest/download/install.sh | bash
```

### Verify-then-run (не доверяй — проверь)

`curl | bash` означает запуск кода, который ты не читал. Лучше так — скачать, сверить
сумму, прочитать, потом запустить:

```bash
base=https://github.com/Di-kairos/securetrash/releases/latest/download
curl -fsSLO "$base/install.sh"
curl -fsSLO "$base/SHA256SUMS"
shasum -a 256 -c SHA256SUMS --ignore-missing   # проверяет install.sh
less install.sh                                  # прочитать глазами
bash install.sh
```

> **Целостность ≠ подлинность (честные границы).** Контрольная сумма доказывает, что
> скачанный бинарь совпадает с `SHA256SUMS` из **того же релиза** — это ловит
> повреждение, частичную/кэш-подмену и не даёт запустить код с подвижной ветки `main`.
> Она сама по себе НЕ защищает от атакующего, способного переписать *и* бинарь, *и* его
> сумму в источнике (или на твоём канале), и НЕ доказывает, *кто* их опубликовал. Для
> этого нужна подпись (notarization / Authenticode — ведётся отдельно, F-4) или
> **Homebrew**, чей ожидаемый хеш лежит в git-истории tap'а, а не рядом со скачиваемым
> файлом. Для воспроизводимости фиксируй версию: `ST_VERSION=0.4.12` вместо `latest`.

### Windows (beta)

PowerShell-порт лежит в [`windows/`](windows/README.md). Та же честная логика на
**BitLocker** (аналог FileVault) и crypto-shred через BitLocker-VHDX, с **VeraCrypt**
как fallback на редакциях без BitLocker. Установщик проверяет `securetrash.ps1` по
`SHA256SUMS` с релизного тега до установки.

```powershell
irm https://github.com/Di-kairos/securetrash/releases/latest/download/install.ps1 | iex
```

> **Beta:** Windows-порт протестирован по логике (Pester, замоканные Windows-API), но
> ещё не проверен на реальном Windows-железе. См. [`windows/README.md`](windows/README.md).

## Быстрый старт

Сценарий с папкой-корзиной — для повседневного удаления:

```bash
securetrash setup          # создаёт ~/SecureTrash, alias sectrash, проверяет FileVault
securetrash check          # честный аудит: FileVault + тип диска + вердикт
# перетащи ненужные файлы в ~/SecureTrash
securetrash empty          # опустошить папку (на HDD — перезапись, на SSD — см. ниже)
```

Сценарий с зашифрованным контейнером — для секретов (реальная гарантия на SSD):

```bash
securetrash vault create   # создаёт ~/SecureVault.sparsebundle (AES-256), спросит пароль
securetrash vault open      # монтирует контейнер в /Volumes/SecretVault
securetrash vault status    # открыт сейчас контейнер или закрыт?
# работай с секретами внутри /Volumes/SecretVault
securetrash vault close     # размонтировать — данные снова зашифрованы
securetrash vault destroy   # уничтожить контейнер + ключ (crypto-shred, необратимо)
```

## Команды

| Команда | Что делает |
|---|---|
| `securetrash check` | Аудит FileVault и типа диска, честный вердикт о гарантиях |
| `securetrash setup` | Создаёт `~/SecureTrash`, ставит alias `sectrash`, предупреждает если FileVault выключен |
| `securetrash empty` | Опустошает `~/SecureTrash` (best-effort перезапись; на SSD **не** гарантия) |
| `securetrash shred <path>...` | Удаляет файл или папку (best-effort; на SSD **не** гарантия — см. `check`) |
| `securetrash vault create\|open\|close\|destroy\|status` | Зашифрованный контейнер (AES-256) для crypto-shred |
| `securetrash vault reset [size]` | По-настоящему очистить сейф: crypto-shred содержимого + пересоздание пустого (сам сейф остаётся) |
| `securetrash version` | Показывает версию |

## Как это работает

В основе vault лежит **crypto-shred** — стирание через уничтожение ключа.

Контейнер (`sparsebundle`) шифруется AES-256 с самого момента создания: всё, что
попадает внутрь, пишется на диск уже как шифротекст. Открытый контейнер
монтируется как обычный том `/Volumes/SecretVault`; закрытый — это нечитаемый
зашифрованный образ.

Когда ты делаешь `vault destroy`, удаляется сам контейнер вместе с ключом
шифрования. Дальше восстановление зависит от стойкости твоего пароля и от того,
что нигде не осталось копий, бэкапов или снимков контейнера. При стойком пароле
и без оставшихся копий блоки на SSD — это, по сути, математический шум: их можно
сколько угодно вычитывать утилитами восстановления, осмысленных данных там нет.

Это и есть преимущество над перезаписью: тебе **не нужно** дотягиваться до
конкретных физических ячеек (что на SSD невозможно из-за wear leveling и TRIM).
Достаточно уничтожить один маленький ключ — и весь объём данных становится
бессмысленным разом.

## FAQ

**Можно ли восстановить файл после `shred`?**
На HDD `shred`/`empty` делает проходы перезаписи — это best-effort и обычно
помогает, но всё равно **не гарантия** (нет контроля над bad/remapped-секторами).
На SSD перезапись тоже **не гарантия** (wear leveling, COW, TRIM). Поэтому
для секретов полагайся не на `shred`, а на FileVault + `vault`.

**Почему `vault`, а не перезапись?**
Перезапись пытается затереть конкретные ячейки, но на SSD ты физически не
управляешь тем, куда контроллер пишет. Crypto-shred обходит проблему: данные
изначально зашифрованы, и уничтожение ключа делает их нечитаемыми независимо от
того, где блоки лежат.

**Работает ли это без FileVault?**
Базовой защиты не будет. Без FileVault «удалённые» блоки на SSD могут оказаться
восстановимыми. Включи FileVault: System Settings → Privacy & Security →
FileVault. `securetrash check` это проверит.

**Безопасно ли `vault destroy`?**
Операция **необратима**: контейнер и его ключ удаляются навсегда. Поэтому команда
запрашивает явное подтверждение (нужно ввести `yes` или передать `--yes` в
скриптах). После неё восстановление зависит от стойкости пароля и от того, что
нигде не осталось копий/бэкапов/снимков контейнера.

**Защищает ли открытый vault то, что внутри него?**
Только в состоянии покоя. Пока vault *смонтирован*, его plaintext всё ещё может
утечь в swap, индекс Spotlight, Time Machine или облачную синхронизацию. Для
секретов высшей ценности (seed-фраза кошелька, приватный ключ) crypto-shred-vault —
правильное место, чтобы **хранить и необратимо уничтожить**, но он не заменяет
офлайн / air-gapped обращение, пока секрет реально открыт на работающей машине.

## Scope & limitations

Честность — весь смысл этой утилиты, поэтому вот что она точно **не** делает:

- **Стойкость crypto-shred — условная.** `vault destroy` удаляет контейнер и его
  ключ, но надёжность стирания зависит от **стойкости пароля** и от того, что нигде
  не осталось **копий, бэкапов или снимков** контейнера (Time Machine, облачная
  синхронизация, ручные копии). Слабый пароль или забытый бэкап всё сводят на нет.
- **Содержимое смонтированного vault может утечь.** Пока контейнер открыт, его
  plaintext-содержимое может быть проиндексировано или скопировано **Spotlight**,
  записано в **swap**, попасть в **Time Machine** или уйти в **облачную синхронизацию**.
  SecureTrash эти места не вычищает.
- **Перезапись — best-effort, а не гарантия.** На SSD/APFS гарантии стирания нет
  (wear leveling, COW, TRIM); даже на HDD она не достаёт до bad/remapped-секторов.
- **FileVault — фундамент.** Без полнодискового шифрования «удалённые» блоки могут
  оказаться восстановимыми. Утилита предупредит, но включить FileVault за тебя не может.

## Дисклеймер

Программа предоставляется «как есть», без каких-либо гарантий (см. [LICENSE](LICENSE)).
Безопасность зависит от твоего окружения:

- Проверь, что FileVault включён: `fdesetup status` (должно быть `FileVault is On.`).
- `vault` — **превентивная** мера: он защищает данные, созданные внутри него,
  и не стирает задним числом то, что уже лежало на диске незашифрованным.
- На SSD/APFS перезапись отдельных файлов не даёт гарантий стирания — это
  фундаментальное свойство технологии, а не ограничение этой утилиты.
